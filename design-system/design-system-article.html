Design systems are products, not projects

Notes on tokens, trust, and the discipline that keeps a growing product from quietly falling apart.

Most teams don't decide to abandon their design system. It happens a component at a time — a one-off button here, a slightly different spacing there — until the system is a folder of good intentions and the product is a patchwork held together by whoever shipped last. This is a short field guide to the parts that actually keep a system alive.

Why systems quietly drift

A design system rarely fails with an announcement. It fails one exception at a time: a deadline that can't wait for a new variant, a designer who eyeballs a spacing value instead of pulling it from the scale, an engineer who hardcodes a hex because the token wasn't exposed yet. Every exception is reasonable on its own. The system erodes anyway, because nothing is ever consciously deciding to break it — the drift is the sum of a hundred small, defensible decisions.

The fix isn't more rules. Teams that police every deviation end up with a system nobody wants to touch. The fix is making the system the easiest path, not the most correct one on paper. If pulling a token is faster than typing a value, people pull the token. Convenience is the real enforcement mechanism.

Start with tokens, not components

It's tempting to start a design system with the components everyone can see — buttons, cards, modals. But components built on hardcoded values are just as brittle as the ad-hoc UI they replace; they're simply brittle in a shared file now. Tokens are what make a system a system: a single source for color, spacing, radius, and type that every component reads from instead of guessing.

"A component library without tokens is a set of opinions. A token layer without components is just a spreadsheet. You need both, but tokens have to come first."

— on sequencing a system

In practice this means naming things by role, not by appearance. color.accent.600 survives a rebrand. color.indigo.600 does not — the day the brand color changes, every name becomes a lie. Below is a minimal token file that most systems can grow from:

/* tokens.css — the layer everything else reads from */
:root {
  --color-accent-600: #3730A3;
  --color-ink-900: #14141A;
  --space-4: 1rem;
  --space-6: 1.5rem;
  --radius-md: 10px;
  --font-body: "Inter", sans-serif;
}

A component is a promise

Publishing a component to a shared library is a commitment, not a convenience. Every team that adopts it is trusting that its states are complete, its accessibility is handled, and its API won't change without warning. Systems that treat components as promises version them deliberately and document what "stable" means. Systems that treat them as convenience end up with three slightly different buttons and no way to tell which one is safe to use.

Documentation is an interface too

Nobody reads a design system doc for pleasure. They read it because they're stuck, mid-task, trying to ship something. That means documentation has the same job as any other interface: get someone unstuck as fast as possible. The best docs lead with the decision ("use this when…"), not the API surface. The API is for after the decision is made.

  • Show the component in the states people actually hit — loading, error, empty — not just the happy path.
  • Say what not to use it for. Negative guidance saves more time than positive examples.
  • Keep one canonical example per pattern. Three examples that quietly disagree are worse than one.

Governance without bureaucracy

Governance has a bad reputation because most of it is theater — a review board that meets to approve decisions that were already made. Useful governance is closer to a contract: clear rules for who can propose a new token, how a breaking change gets communicated, and what happens when a team needs an exception right now. The goal isn't to slow people down. It's to make sure fast decisions don't quietly become permanent ones.

What "done" looks like

A design system is never finished, but it can be healthy. Healthy looks like: new screens getting built faster than they were a year ago, designers spending less time debating spacing and more time on the actual problem, and a support queue that isn't full of "why does this button look different here" tickets. If you're not seeing that, the system isn't the thing that's broken — the incentives around using it usually are.

Treat the system like the product it is: give it an owner, a changelog, and users you actually listen to. Everything else in this piece is just detail in service of that one idea.
