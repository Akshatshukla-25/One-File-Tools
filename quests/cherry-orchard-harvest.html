📖 The Orchardist's Field Guide

Everything the harvest game is teaching you, written down in plain git.

🍒 Picking a single cherry

Cherry-picking takes the exact change introduced by one commit on another branch, and replays it onto your current branch as a brand-new commit.

# apply just this one commit onto your current branch
git cherry-pick a1b2c3d

Nothing else from that branch comes along — no other commits, no merge history. Just that one diff.

🧺 Picking several at once

# pick three specific commits, oldest to newest
git cherry-pick a1b2c3d e4f5a6b c7d8e9f

# or pick a contiguous range (exclusive of the first hash)
git cherry-pick a1b2c3d^..c7d8e9f

⚠️ When the fruit doesn't fit: conflicts

If a commit changes a line that depends on context your current branch doesn't have yet — like a fix that assumes an earlier fix is already applied — git can't automatically place it. That's a conflict, and it usually means you're picking out of order.

# after fixing the conflicted lines by hand:
git add <file>
git cherry-pick --continue

# or bail out entirely and go back to before you started:
git cherry-pick --abort
--continue--abort--skip

🌳 Cherry-pick vs. merge vs. rebase

Cherry-pick copies chosen commits as new commits with new hashes — the source branch is untouched, and only your handpicked changes travel.

Merge brings every commit from the other branch across at once, plus a merge commit tying the histories together.

Rebase replays an entire branch's commits, in order, on top of another — also producing new hashes, but for the whole branch, not a subset.

pick = a few, new hashesmerge = all, +merge commitrebase = all, replayed

🚜 Why orchardists cherry-pick in real life

The classic case is a hotfix: a critical bug fix lands on your main development branch, but you need it on an already-released version too, without dragging along unfinished features that also live on main. Cherry-pick that one fix commit onto the release branch and ship it.
